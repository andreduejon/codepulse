# v0.5.0 Plan

Goal: add repo-group switching and a debug dialog without expanding write scope.

Status: implemented for v0.5.0. Final behavior differs from early sketches where
noted below.

## Why this release

`v0.4.0` added multiple read-only providers inside one repo. Next friction point is
switching between related repos and understanding what the app is doing when git
or provider requests feel slow.

`v0.5.0` should improve:
- fast switching between related repos in one logical app family
- visibility into executed git commands and provider requests
- mode clarity by removing mode-cycling from `shift + ←/→`

This release is about navigation and observability.

## Release shape

### Phase 1 — repo grouping
Add repo grouping metadata and a compact command-row project carousel for fast
switching between related repos.

### Phase 2 — debug dialog
Add `:debug` read-only dialog showing recent git/provider activity and errors.

## Non-goals

- no write actions
- no cross-repo unified graph
- no repo auto-discovery from filesystem
- no group UUIDs
- no drag/drop sorting
- no group-wide reload/fetch actions
- no filtering or clearing inside debug dialog beyond existing global `:clear`
- no persisted debug history

## Repo grouping

### Config shape

Store group metadata per repo entry:

```json
{
  "repos": {
    "/abs/path/frontend": {
      "group": "e-ant",
      "appName": "e/ant Frontend"
    },
    "/abs/path/backend": {
      "group": "e-ant",
      "appName": "e/ant Backend"
    }
  }
}
```

Rules:
- `group` is free-text editable
- `appName` is free-text editable
- no dropdown for existing groups
- no UUID; group string is canonical key
- `appName` falls back to repo basename when unset
- missing / empty / whitespace-only values are treated as unset
- store optional values as omitted/undefined, not empty string placeholders

Validation:
- `group`: max 64 chars
- `appName`: max 64 chars
- general text fields: max 255 chars
- URL fields: max 2048 chars
- path-like fields: max 2048 chars
- trim whitespace before validation
- reject invalid values rather than silently truncating

### Menu UX

Repository tab should expose:
- `Group` editable field
- `App name` editable field

No suggestion dropdown. User types exact values.

If values are too long or invalid:
- keep editing state visible
- show invalid marker/message in menu
- do not persist invalid value

### Project switcher grouping

The project switcher should cluster repos by `group` as well.

UI rules:
- group heading shown as muted plain text
- no indentation for repo rows under a group
- one blank line between groups
- ungrouped repos appear under muted `ungrouped`
- keep one flat navigable list; no collapse in `v0.5.0`

Example:

```txt
e-ant
e/ant Frontend
e/ant Backend
e/ant Microservice

platform
API Gateway
Auth Service
```

Sorting:
- groups sorted alphabetically
- repos within a group sorted alphabetically by `appName || basename`

### Command-row project carousel

Show group members only when current repo has a group and at least one sibling.

Placement:
- integrated into the command/status row after provider + mode
- stable single-line layout only (no wrapping)
- branch badges stay right-aligned

Selected repo:
- accent color
- branch-label style chip
- format: `· e/ant Frontend ·`

Other group members:
- muted badge style
- format: `e/ant Backend`

Overflow markers:
- muted
- left: `◂3`
- right: `2▸`

Overflow policy:
- single line only
- window list around selected repo
- selected repo always visible
- markers appear only when items are hidden off-screen

### Navigation

`shift + ←/→` switches between group members.

Rules:
- does not wrap at group ends
- no remote fetch on switch
- cold repo switch: load local repo data only
- warm repo switch: restore cached session state instantly where possible
- switch immediately; do not wait for active work to finish
- active dialogs close on switch
- ancestry/path/search/command modes exit back to normal mode on switch

Preserve on switch:
- provider view if available
- per-repo cached provider data
- repo-local graph/cache snapshot
- loaded commits/page state
- cursor position when valid
- graph scroll position when valid

Reset on switch:
- dialogs
- command/search/path/ancestry active mode
- transient status messages

Missing-state behavior on restore:
- if previous cursor target no longer exists, clamp to next valid item
- if no valid target remains, fall back to top

In-flight work:
- abort active git/provider requests when switching repos if possible
- do not block switch waiting for completion

### Reload/fetch behavior

Do not add group-wide reload/fetch in `v0.5.0`.

Rules:
- `f` fetches current repo only
- `:reload` reloads current repo only
- switching group members never auto-fetches remote

## Repo session cache

Use in-memory per-repo session cache only.

Shape:
- key: absolute repo path
- value: repo session snapshot

Stored in snapshot when practical:
- commits / graph rows
- branches / current branch / remote URL
- cursor position / scroll position
- active provider view
- graph badges
- provider status / last successful refresh
- provider-specific in-memory caches

Not stored:
- dialogs
- command/search/path/ancestry active mode
- transient input drafts
- debug dialog open state
- provider internal API caches

Cache policy:
- restore cached snapshot immediately on warm switch
- cold switch does normal local load
- no disk persistence in `v0.5.0`

## Debug dialog

### Command

Add:

```txt
:debug
```

Behavior:
- toggles debug dialog open/close
- dialog closes with `esc`
- no extra actions inside dialog

### Data shown

Keep last 100 events in memory.

Ordering:
- newest first
- long lines truncated in the list view

Event types:
- git commands
- provider HTTP requests
- errors

Optional later, but not required now:
- cache hits/misses

Each event should include when possible:
- timestamp
- source (`git`, `GitHub`, `Jenkins`, `error`)
- command or sanitized path
- result/status
- duration

Example:

```txt
12:41:03 git     git log --all --max-count=200               ok     84ms
12:41:05 GitHub  POST /graphql                               200   312ms
12:41:08 Jenkins GET  /job/foo/api/json?...                  200   918ms
12:41:10 error   Jenkins auth failed: SSO login required
```

### Safety

Must redact:
- auth headers
- tokens
- env var values
- signed URLs
- sensitive query params

If easy to implement in first pass, replace hidden values with `********`.

### Clear behavior

`:clear` should clear:
- current error/info state
- debug event buffer

No clear command inside dialog.

## Config hardening

Menu validation alone is not enough because users can paste values directly into
the config file.

On config load:
- validate the same length limits used by the menu
- ignore invalid values with warnings
- never crash on oversized strings
- never silently accept a huge `group` / `appName` value into active UI state

Examples:
- oversized `group` or `appName` -> ignore field, warn, fall back safely
- oversized Jenkins username/token env/host -> ignore field, warn
- oversized URL/path fields -> ignore field, warn

## Keyboard model cleanup

Make mode entry explicit:
- ancestry only `a`
- path only `p`
- command only `:`
- search only `/`

Repurpose `shift + ←/→` fully for group switching.

Remove old mode-cycling behavior.

## Acceptance criteria

### Grouping
- repo config supports `group` and `appName`
- command-row project carousel only appears for grouped repos with siblings
- selected repo badge uses accent style
- non-selected repos use muted badge style
- overflow markers appear when list exceeds width
- `shift + ←/→` switches repos without wrapping and never auto-fetches remote

### Debug
- `:debug` toggles a debug dialog
- dialog shows up to 100 recent events
- events include git commands, provider requests, and errors
- all sensitive values are redacted
- `esc` closes dialog
- `:clear` clears debug buffer and current status/error state

### Navigation cleanup
- `shift + ←/→` no longer cycles modes
- `a`, `p`, `:`, `/` remain sole entrypoints for ancestry/path/command/search
