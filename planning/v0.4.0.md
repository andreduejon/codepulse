# v0.4.0 Plan

Goal: prepare provider system for more integrations, then add Jenkins MVP.

## Why this release

`v0.3.0` proved GitHub Actions as read-only provider. `v0.4.0` should make provider work repeatable. Main target:
- stop hardcoding provider-specific glue in app layer
- make provider config, auth, state, cache, and help content reusable
- add Jenkins as first new provider using same platform

This release is about foundations first, feature second.

## Release shape

### Phase 1 — provider platform cleanup
Build shared layer for all providers:
- config shape
- state shape
- auth/availability shape
- cache shape
- help/menu docs shape
- refresh/error/loading handling
- capability flags

### Phase 2 — Jenkins provider MVP
Add Jenkins read-only provider with:
- host + user + token env config
- full job URL support
- latest build summary
- console log dialog
- branch match
- exact SHA match when metadata exists
- fallback label when only branch-level match exists

## Non-goals

- no write actions
- no Jenkins CLI dependency
- no auto-discovery across every Jenkins folder layout
- no pipeline editing
- no remediation actions
- no Snyk/Sonar/Trivy in this release

## User story

Example:

1. user opens `codepulse` on repo
2. provider tab shows Jenkins status next to GitHub Actions
3. selected commit has matching Jenkins build
4. user opens build log in dialog
5. user sees summary + build output without leaving TUI

Fallback example:

- exact SHA not found
- app shows `latest build for branch main`
- app does not pretend exact commit match

Missing auth example:

- `JENKINS_TOKEN` not set
- provider shows unavailable state
- help text says which env var is missing

## Phase 1 details

### 1. Provider registry

Current provider logic should move toward explicit provider definitions.

Suggested shape:
```ts
type ProviderId = "github-actions" | "jenkins";

interface ProviderCapabilities {
  summary: boolean;
  detail: boolean;
  logs: boolean;
  exactSha: boolean;
  branchFallback: boolean;
}

interface ProviderDefinition<Config> {
  id: ProviderId;
  label: string;
  capabilities: ProviderCapabilities;
  loadConfig(config: unknown): Config | null;
  isEnabled(config: Config): boolean;
  isAvailable(config: Config, env: Record<string, string | undefined>): boolean;
}
```

Example:
- GitHub Actions supports summary/detail/logs/exact SHA
- Jenkins supports summary/detail/logs/branch fallback, exact SHA if metadata exists

### 2. Shared status model

Unify provider states:
- `idle`
- `loading`
- `unavailable`
- `error`
- `ready`

Example UI mapping:
- `unavailable` -> setup hint in selector/menu
- `error` -> muted error lane above footer
- `loading` -> spinner + short label

### 3. Shared auth/availability model

Providers should report *why* unavailable.

Examples:
- missing env var
- disabled in config
- unsupported host
- missing job URL
- no matching project/job

Example message:
```txt
Jenkins unavailable: missing JENKINS_TOKEN
```

### 4. Shared cache model

Use local cache files for fetched provider results.

Keys should include:
- repo root
- provider id
- branch or sha query key
- provider-specific binding id

Example cache entry:
```json
{
  "repo": "/Users/me/repo",
  "provider": "jenkins",
  "query": "main@abc123",
  "fetchedAt": "2026-05-04T12:00:00Z",
  "summary": { "success": 1, "failed": 0 }
}
```

### 5. Shared docs/help model

Help/menu text should come from provider metadata, not hand-written copy in multiple files.

Example:
- provider config section in menu
- provider setup section in help dialog
- provider CLI help section

## Phase 2 details — Jenkins MVP

### 1. Config

Recommended config shape:
```json
{
  "providers": {
    "jenkins": {
      "enabled": true,
      "host": "https://jenkins-ti.app.lhtcloud.com",
      "username": "ci-user",
      "tokenEnvVar": "JENKINS_TOKEN",
      "jobs": [
        {
          "label": "prepare-release",
          "url": "https://jenkins-ti.app.lhtcloud.com/job/e-ant/job/jobs/job/prepare-release/"
        }
      ]
    }
  }
}
```

Notes:
- never store raw token in config
- full job URL is canonical input
- label is for UI only

### 2. Job discovery

Do not start with path parsing UI.

Use full URLs because Jenkins layouts vary:
- nested folder jobs
- multibranch jobs
- org folders
- mixed naming schemes

Example URLs:
```txt
https://jenkins-ti.app.lhtcloud.com/job/e-ant/job/jobs/job/prepare-release/
https://jenkins-ti.app.lhtcloud.com/job/e-ant/frontend-multi-branch-pipeline
```

### 3. API endpoints

Use REST endpoints only.

Examples:
```txt
<jobUrl>/api/json
<jobUrl>/<buildNumber>/api/json
<jobUrl>/<buildNumber>/consoleText
```

### 4. SHA mapping

Goal: map build to commit when possible.

Exact match sources:
- `GIT_COMMIT`
- `commitId`
- `SHA1`
- SCM revision action fields

Fallback sources:
- branch name
- latest build for branch

Example exact match:
```txt
selected commit abc123 -> build #92 -> exact match
```

Example fallback:
```txt
selected commit abc123 -> no exact SHA metadata -> latest build for branch main
```

### 5. Logs

Jenkins log support should show:
- console text
- build status
- duration
- build number

Example dialog title:
```txt
prepare-release #92 · SUCCESS · 2m 13s
```

### 6. UI placement

Suggested surfaces:
- provider tab in graph
- detail panel summary row
- build log dialog
- menu/help provider setup text

Example summary row:
- `SUCCESS · build #92 · branch main · exact commit match`

### 7. Errors

Errors must be explicit.

Examples:
- missing token env var
- 401 unauthorized
- 404 job not found
- 403 forbidden
- no build for selected sha
- no exact sha metadata in build

Do not collapse all cases into `No data`.

## Implementation order

1. extract provider definition/state/capability model
2. move GitHub provider onto shared model
3. wire shared help/menu text generation
4. add Jenkins config schema
5. add Jenkins API client
6. add Jenkins provider UI
7. add tests for URL normalization, auth, sha extraction, fallback logic

## Example tests

### URL normalization
Input:
```txt
https://jenkins.example.com/job/a/job/b/
```

Expected:
- trailing slash normalized
- `/api/json` appended cleanly

### SHA extraction
Input build JSON contains:
```json
{ "actions": [{ "lastBuiltRevision": { "SHA1": "abc123" } }] }
```

Expected:
- extracted sha = `abc123`

### Branch fallback
Input build JSON contains branch name but no sha.

Expected:
- build can still be shown
- label says branch latest, not exact commit

### Missing auth
No `JENKINS_TOKEN` env.

Expected:
- provider unavailable
- help text mentions missing env var

## UX rules

- exact match first, fallback second
- never lie about commit mapping
- never store raw secrets in repo config
- keep provider setup readable in TUI
- keep log text paged and stable

## Success criteria

`v0.4.0` is done when:
- provider system is reusable for new integrations
- Jenkins provider works against real company job URLs
- exact SHA mapping works when Jenkins exposes it
- fallback branch/latest view works when it does not
- logs are readable in TUI
- tests cover URL/auth/sha/fallback paths

## Locked decisions

These are now fixed for `v0.4.0`:

- scope = provider framework + Jenkins MVP
- keep current provider registry, add adapter layer; no rewrite
- one normalized result shape for all providers
- cache = in-memory only
- config stays in existing app config file
- secrets via env var names only; no OAuth/SSO
- provider order = fixed by config
- provider views remain separate tabs/views
- Jenkins config = flat `jobs[]` list
- Jenkins source of truth = full job URL
- exact SHA only for first cut
- no fake branch/latest fallback for core commit match
- empty commit cell = `......` when no job present
- detail history keeps same-SHA runs only
- add/remove config UI is local only; no Jenkins writes
- stages/steps are optional and only loaded in exact detail view if API access is cheap

## Acceptance criteria

### Provider framework

- app can register more than one provider without special-case wiring in UI
- provider availability, errors, and loading states use shared model
- GitHub provider still works through new shared layer
- provider menu/help text comes from shared metadata where possible

### Jenkins provider

- user can add Jenkins job by full URL
- user can optionally set label for job display
- Jenkins provider reads token from env var ref, not raw secret in config
- overview shows job tree / grouped structure with run badges
- exact commit with no job shows `......`
- detail view shows same-SHA history only
- stages/steps appear only when exact job detail loads and only if API supports them cheaply

### UX / behavior

- `Tab` cycles providers in config order
- separate provider tabs remain visible
- no misleading fallback match shown as exact commit match
- config add flow is local-only and read-only to Jenkins

## Open check before build

If yes, include them in detail view.
